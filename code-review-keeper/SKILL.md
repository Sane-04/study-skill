# 必敲代码复习管理（Code Review Keeper）

## Overview

本 skill 负责"必敲代码"的长期记忆管理：把学习中标为🔴必敲的核心代码归档，并按艾宾浩斯遗忘曲线安排复习。复习靠用户**按需触发**（说"复习"），不依赖定时任务；漏复习的条目会自动累积为逾期项，优先提醒。

## 复习档案位置（保持可迁移）

- 默认档案：**用户桌面下的 `必敲代码复习.md`**。
- **不要在 skill 中写死任何绝对路径**，以保证 skill 可被任何用户复用。
- 首次使用时，先确认/解析当前用户桌面的实际路径，再读写该文件；若文件不存在则按下方"档案结构"创建。

## 档案结构（Markdown）

文件顶部放说明 + 艾宾浩斯规则，正文每段必敲代码一个条目。条目用如下字段（用清单或小标题组织，保证人和 AI 都易读）：

| 字段 | 含义 |
|---|---|
| 编号 | 递增序号，如 001 |
| 主题 | 这段代码属于什么，如"Self-Attention 单头" |
| 首学日期 | 第一次学习/敲的日期（YYYY-MM-DD） |
| 复习进度 | 当前到第几次复习：R0(未复习)~R6 |
| 下次复习日 | 按艾宾浩斯算出的日期（YYYY-MM-DD） |
| 骨架口令 | 一句话概括代码逻辑链，用于主动提取，如"QKV投影→打分除√d→下三角填-inf→softmax→乘V" |
| 代码 | 完整的必敲代码块 |

## 艾宾浩斯复习节奏

以"首学日期"为基准，复习间隔（天）：

| 复习点 | 距首学天数 | 含义 |
|---|---|---|
| R1 | 1 | 第 1 天 |
| R2 | 2 | 第 2 天 |
| R3 | 4 | 第 4 天 |
| R4 | 7 | 第 7 天 |
| R5 | 15 | 第 15 天 |
| R6 | 30 | 第 30 天 |

`下次复习日 = 首学日期 + 当前复习点对应天数`。R6 完成后视为长期掌握，可标记"已毕业"，不再排复习。

## 三类核心操作

### 操作一：归档一段必敲代码（追加条目）
当被要求记录一段必敲代码（通常由 study-tutor 讲完🔴必敲代码时触发，或用户直接要求）：
1. 读取/创建桌面 `必敲代码复习.md`。
2. 生成新条目：编号递增、主题、首学日期=今天、复习进度=R0、下次复习日=今天+1（即 R1）、骨架口令、完整代码。
3. 追加到档案，向用户确认"已归档第 NNN 条 + 下次复习日"。

### 操作二：判定今天该复习哪些 + 出题考核
当用户说"复习/该复习哪些了/考考我"：
1. 读档案，取所有 `下次复习日 ≤ 今天` 的条目为"待复习"，逾期越久越靠前排序。
2. 若数量多，每次最多列 5 条，避免压垮；其余提示"还有 N 条逾期，复习完可继续"。
3. **用主动提取方式考核，不直接贴出完整代码**：
   - 先只给"主题 + 骨架口令"，要求用户从零重建/默写或口头讲解；
   - 用户尝试后，再对照档案中的代码逐点点评对错、补关键细节。
4. 用一张表展示本次待复习清单：

| 编号 | 主题 | 首学日期 | 复习进度 | 逾期天数 |
|---|---|---|---|---|

### 操作三：根据反馈更新复习进度
用户对某条给出反馈后：
- **"过了/熟/记得"** → 复习进度推进到下一个 R，按艾宾浩斯重算"下次复习日"，写回档案。
- **"不熟/忘了/没记住"** → 复习进度重置为 R1（下次复习日=今天+1），从头再走一轮间隔。
- 每次更新后，简要回报该条新的"复习进度 + 下次复习日"。

## 反面清单（不要做）

- 不要在 skill 或档案逻辑里写死绝对路径，必须保持可迁移。
- 不要在考核时一上来就贴出完整代码，应先让用户主动重建/讲解。
- 不要把没到期的条目也拿来复习（除非用户主动要求）。
- 不要在用户反馈"不熟"后仍推进复习进度，应重置为 R1。
- 不要把本 skill 当作讲解新知识的工具（那是 study-tutor 的职责）。
